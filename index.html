A few notes on starting with React 📓

A few things to watch for when first starting to develop with React.

These took me by surprise and I had to get use to writing this way, but it wasn't too long before I got used to it.

And if you do make a mistake and write something incorrectly, you'll get a friendly warning in your browser dev tools console.

👇

camelCase

Write attributes in camelCase 🐪

  • tabindex 👉 tabIndex
  • contenteditable 👉 contentEditable
  • maxlength 👉 maxLength

(aria-* and data-* are exempt from this.)

When it comes to writing HTML attributes in React components, they need to be written in camelCase style.

This means that when an attribute name is made up of two words, the second word in the attribute must start with a capital letter.

👇

Reserved words

When HTML conflicts with JS 😕

  • for 👉 htmlFor
  • class 👉 className

There are a few reserved word conflicts when writing HTML attributes in React components.

For example…

  1. The reserved word "for" is used in HTML when pairing a "label" with an "input" element. In JavaScript, "for" is used to loop through items using an index. So the "for" attribute needs to be written out as, "htmlFor."
  2. Another example, the reserved word "class" is used in HTML to attach CSS selectors to apply visual styles to elements. In JavaScript, it's used to create a class declaration. So when adding a class to an element, this needs to be written as, "className."
👇

Self closing tags require /

What is this, XHTML? 🤷‍

  • <img src="…" alt=""> 👉 <img src="…" alt="" />
  • <meta charset="utf-8"> 👉 <meta charset="utf-8" />
  • <input type="text"> 👉 <input type="text" />

In HTML5 there are a few elements which do not feature a closing tag.

Just like back in the old XHTML days, JSX (which is a sub-set of JavaScript used to write React components) requires self closing elements to have the forward slash.

Setting the page <title> 🗂

Now, let's take a look at how to set the page "title."

👇

Why bother? 🤔

  • Updates the browser tab
  • SEO
  • Often the first content announced by screen readers

So, why bother with setting the page "title" content?

Well, a few reasons:

  1. It updates the content in the browser tab, so sighted users have a better understanding of where they are< in the app/li>
  2. It helps to increase page Search Engine Optimization when search engines crawl and index your site
  3. And the page "title" content is often the first bit of information announced by screen readers, so users of assistive technology will have a better understanding of where they're currently at in the app as well

So, how do we do this in React?

👇

Use document.title

            
              componentDidMount() {
                document.title = 'My page title';
              }
  • Not specific to React
  • More on MDN

The simplest method that I found is to set the "title" using the JavaScript "document.title" property.

On the screen here is a function called "component did mount." This is a React lifecycle method. The code within this function executes when the compomnent is loaded onto the screen.

The single line in this function is using the "document.title" property to set a string value of, "My page title" as the title. So when this components loads, the page "title" will be set to this value.

This is not specific or unique to React; this method can be used in any JavaScript based application or site.

Live announcements 📢

Now let's take a look at a complete, live announcements component.

Sometimes when you're working with a single page app, you request data from an external data source.

For people who rely on assistive technology it's a good idea to announce if the data request was successful, if there were any errors, or if successful for example, how many items are now on displayed the screen as a result.

Let's take a look at how to create a live announcement component.

👇

The Announcements component

            
                import React from 'react';

                class Announcements extends React.Component {
                  render() {
                    return (
                      <div aria-live="polite" aria-atomic="true" className="visuallyhidden">
                        {this.props.message}
                      </div>
                    );
                  }
                }

                export default Announcements;

Here we have the "announcements" component in its entirety.

(Explain what's on the sreen…)

Now that we have the component writen and available, how do we use it?

👇

Usage, part 1

Create a state property

            
              this.state = {
                message: null
              };

So, how do we use this component?

To start, in the parent component which will use the "announcements" component, we'll create a new "state" property.

We'll call this property, "message," and set its value to "null" for now.

So, on the screen there's a "this.state" object declaration, and inside the object is a property called, "message" with its default value set to "null."

👇

Usage, part 2

Set the state property

            
              this.setState({
                message: 'Some message'
              });

Next, at the point in your app when you recieve your data, either successfully or if things didn't work out as expected, we'll set the state of the "message" property.

To do so, you call the "this.setState" function and pass in an object parameter.

This object will feature the "message" property we created earlier, with its value set to the string of text we want to announce.

👇

Usage, part 3

Send message to <Announcements /> component

            
              <Announcements message={this.state.message} />

Lastly, we'll include the "announcements" component within the parent component.

In order to send it the message text from state to to the "announcements" component, we use a prop.

The prop on the screen here is called, "message" with its value being set to whatever's currently in the "message" state property.

So when this component loads, the message will be announced by a screen reader and the person using the app will be infomred of the current state.

👇

Focus management 🔀

Well planned focus management is critical in creating a successful and comfortable user experience, especially for highly dynamic apps.

What I mean by focus management is moving the keyboard cursor from one place to another in order to assist the person on the other side of the screen in moving along with the intended flow of the app.

Let's take a look at a couple examples.

👇

Selecting an element

Kind of like jQuery 🤷‍♂️.

            
              var loadingMessage = $('#loadingMessage');
              // …
              loadingMessage.focus();

First thing we need to do is select an element to send focus to. How do we do this in React?

Well, it can be helpful to think about how you might be used to selecting an element with something like jQuery.

On the screen is an example of the jQuery dollar selector, caching a reference to the "loading message" element in a variable.

Later on in that app, we can use the variable and call its "focus" method to set the keyboard cursor directly onto this element.

👇

The React way

Use a function ref to reference the element elsewhere in the component.

            
              <div
                ref={
                  (loadingMessage) => {
                    this.loadingMessage = loadingMessage;
                  }
                }
                tabIndex='-1'
                >
                Loading…
              </div>

In React, tjhe concept's pretty much the same, but how we arrive there is a little different.

In order to select an element, "the react way," we create what's called a "function ref," or, "ref" for short.

The "ref" is a prop that is set on the element we want to reference later in the component.

(Explain what's on the screen…)

👇

Using the ref…

            
              componentDidMount() {
                this.loadingMessage.focus();
              }

Then in order to use the "ref," we can use the "component did mount" function again, and then call the "focus" method on the "ref" element when then component loads.

When the component loads, the keyboard focus indicator will be automatically sent to this element.

👉

More focus management! 📰

Another pretty important example of focus management is adjusting keybord focus when moving from page to page in React.

👇

Why manage focus when loading a new page? 🤔

When using React Router's <Link /> component, the browser never reloads, leaving the user in an unknown position.

So, why exactly do we need do this?

For one, when using React Router's "link" component to load new page views, the keyboard focus is lost or left in an unpredictable state.

The "link" component dynamically loads new pages into the DOM without reloading the browser. As a result, pages are precieved to load very quickly, but we also need to manage the keyboard focus when this happens.

How do we accomplish this task?

👇

Our new friend, ref

            
              <div
                ref={
                  (contentContainer) => {
                    this.contentContainer = contentContainer;
                  }
                }
                tabIndex="-1"
                aria-labelledby="pageHeading"
                >
                  <Header />
                    <h1 id="pageHeading">…</h1>
                    // Content components…
                  <Footer />
              </div>

Coming back to to the "ref" concept, we can create a "ref" on the container div and use that to shift focus to the page.

So, for this example there's a wrapper "div" container which holds all of the page content, including the header, footer, etc.

When a new page view loads, focus is sent to the top of the page. From there, the user can move forward from this point, discovering content in the same manner as if a full page reload took place.

(Explain what's on the screen…)

So, how do we use the "ref?"

👇

Set "page" focus via ref

            
              componentDidMount() {
                this.contentContainer.focus();
              }

Again, to use the "ref," we can use the "component did mount" function, and then call the "focus" method on the "ref" element when then component loads.

When the component loads, the keyboard focus indicator will be sent to the "page container" element.

From this point, the user can move on and continue with content discovery.

React's accessibility linter 🛀

React has a really great accessibility linter which is pre-packaged with each app.

As a general note, a code linter is a super handy tool which helps to ensure clean code, or code standards as determined by the team or individual. In other words, they let you know when something's not quite right.

Let's look at a few ways we can use the linter to help ensure a more accessible React app.

👇

eslint-plugin-jsx-a11y

Outputs errors to the browser console automatically.

            
              <img src="images/chrome-console.png" />

The linter in question is called "eslint-plugin-jsx-a11y" and it comes with each app created with the "create-react-app" utility.

By default, without any setup or confiuguration on your part, the linter outputs errors found to the browser dev tools console.

(Describe what's on the screen…)

👇

Code editor a11y linting? 🤔

Let's output errors directly in the editor, too.

Outputting errors to the console is pretty neat, but I think this would be even more useful if these errors were displayed in my editor. Let's see how we can set this up.

(Describe what's on the screen…)

👇

1. Install ESLint Plugin

  • Atom: linter-eslint
  • Sublime Text: SublimeLinter-contrib-eslint
  • VS Code: WebAnalyzer

First install the ESLint plugin for your editor. This is usually done within the editor itself.

On the screen I've got links to some of the more popular editors these days. The names of the plugins include:

  1. "linter dash eslint" for Atom
  2. "Sublime linter dash contrib dash eslint" for Sublime Text
  3. "Web Analyzer" for VS Code
👇

2. Install ESLint && eslint-plugin-jsx-a11y

            
              npm install eslint eslint-plugin-jsx-a11y --save-dev

Next, install ESLint and the "eslint-plugin-jsx-a11y" packages from "npm."

You would do this by opening your terminal and typing:

"npm space install space eslint space eslint dash plugin dash jsx dash a11y space dash dash save dash dev"

This installs these utilities only for your current project, as each project might have a different configuration.

👇

3. Update the .eslintrc file

Add to the "plugins" section:

            
              "plugins": [
                "jsx-a11y"
              ]

Add to the "extends" section:

            
              "extends": [
                "plugin:jsx-a11y/recommended"
              ]

The last thing to do is update, or create if it doesn't exist, your project's "eslint rc" file.

In the plugins section, you'd add "jsx dash a11y." This loads the plugin for use with your editor's ESLint plugin.

In the extends section, you'd add "plugin colon jsx dash a11y forward-slash recommended." This loads the default, recommended rules for the plugin to test your code against.

So, after this is all done, you'd just restart your editor and you should see some errors come up when you've got some accessibility issues in your code!

Writing semantic HTML in React ✍️

The last thing I want to discuss is writing HTML in React.

In the past I've been asked the very broad, open ended question, "How accessible is React?"

This was coming from the knowledge that other "frameworks" had great difficulty when it came to a11y as a result of them providing third-party created user interfaces.

In other words, if something was not accessible, there's not much of a chance of it getting fixed as it relied on a third-party.

👇

React Components are ES6 Classes

Since React uses ES6 classes to make up its components, it’s up to you to continue to write good, clean, semantic HTML.

With React, and other newer JavaScript "frameworks," this less the case.

What I've found is that since React uses ES6 classes to create and export components, it's up to you, the developer and app author, to continue to write valid, semantic HTML.

With the exception of using a third-party module from "npm" it's all up to you. The framework doesn't force you to use something that someone else create, thus lowering the risk of creating an inaccessible product.

👇

In other words…

If there are accessibility issues, it's your fault. This is a good thing! 👍

In other words, if there are any accessibility issues, it's your fault. And this is a good thing! It's up to you as the developer to fix these issues. You don't have to wait for some third-party developer group to maybe get around to fixing these issues someday. You can do it youself, today. Own your product.
